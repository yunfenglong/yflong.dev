10:T535,<h1>Why a terminal metaphor?</h1>
<p>A terminal UI creates a clear narrative: input, response, and iteration. It also reflects how I actually work day to day, which makes the interface personal instead of decorative.</p>
<h2>Architecture choices</h2>
<p>I kept the interaction model and rendering logic separate so each part remains small and testable:</p>
<ul><li><code>hooks/terminal</code> handles state transitions and keyboard interactions.</li><li><code>lib/terminal</code> focuses on command registration and command execution.</li><li><code>components/terminal</code> handles rendering only.</li></ul>
<h2>What worked well</h2>
<ul><li>The command registry pattern made adding commands straightforward.</li><li>A clear visual token system kept the UI consistent across pages.</li><li>Boot-sequence state persisted in <code>sessionStorage</code> prevented repeated friction.</li></ul>
<h2>One practical lesson</h2>
<p>I avoided over-animating the shell. Small transitions are enough when content and response speed are already doing the heavy UX work.</p>
